📌 三、关键问题思考

企业引入数据库运维产品的时机

  • 当数据库数量达到一定规模，人工管理效率低下时
  • 当业务对数据库可用性要求提高，需要自动化监控时
  • 当企业数字化转型，需要专业运维支持时
  • 当运维成本过高，需要降本增效时

数据库工程师的基本运维能力

  • 数据库安装、配置、升级
  • 性能监控与优化
  • 备份恢复与容灾
  • 故障诊断与处理
  • 安全管理与权限控制
  • 自动化脚本编写
📌 四、运维平台与DBA的关系

📋 任务2-1 在Linux系统下安装并配置MySQL

📖 任务说明

MySQL支持多个系统，不同系统下的安装和配置过程不尽相同。考虑到实际应用中的环境多数是Linux系统，本任务要求在Linux CentOS 7下完成安装和配置MySQL 8.0 Community。

🎯 任务目标

  • 掌握MySQL 8.0 Yum Repository源的配置方法
  • 学会使用yum命令安装MySQL
  • 掌握MySQL服务的启动和管理
🔧 步骤1：配置MySQL 8.0安装源

MySQL Yum Repository提供了一种简单、方便的方法，可以使用yum命令来安装和更新MySQL产品。打开Linux CentOS 7，下载MySQL 8.0的Yum Repository源并安装。

# 下载并安装MySQL 8.0 Yum Repository源
sudo rpm -Uvh https://dev.mysql.com/get/mysql80-community-release-el7-3.noarch.rpm

# 安装完成后，会在/etc/yum.repos.d/目录下新增两个文件：
# mysql-community.repo
# mysql-community-source.repo
🔧 步骤2：安装MySQL 8.0

通过yum命令安装MySQL 8.0，系统会列出安装过程中需要使用的插件，在提示中输入"y"后按"Enter"键。

# 使用yum安装MySQL 8.0
sudo yum --enablerepo=mysql80-community install mysql-community-server

# 系统会提示确认，输入 y 继续安装
🔧 步骤3：启动MySQL服务

MySQL 8.0安装完成后，需要启动MySQL服务，否则客户端无法连接数据库。

# 启动MySQL服务
systemctl start mysqld.service

# 查看服务状态
systemctl status mysqld.service

# 如果Active显示active (running)，则表示MySQL服务已经启动
📌 知识扩展：MySQL服务管理命令
# 启动MySQL服务
systemctl start mysqld

# 停止MySQL服务
systemctl stop mysqld

# 查看MySQL服务状态
systemctl status mysqld

# 设置MySQL开机启动
systemctl enable mysqld

⚠️ 问题1：socket文件错误

错误提示

ERROR 2002 (HY000): Can't connect to local MySQL server through socket '/tmp/mysql.sock' (2)

原因分析：MySQL有TCP/IP和Socket两种连接方式。mysql.sock是一个临时文件，在启动MySQL服务后才会生成。如果出现报错，是因为MySQL将其放在/tmp目录下，而Linux系统将其放在/var/mysql目录中。

# 解决方案：创建软链接
sudo mkdir /var/mysql
sudo ln -s /tmp/mysql.sock /var/mysql/mysql.sock
⚠️ 问题2：删除自带的MariaDB

原因分析：Linux CentOS 7自带MariaDB数据库，如果不需要的话，可以考虑将其删除，防止在安装MySQL时出现不可预知的问题。

# 查看并删除MariaDB
rpm -qa | grep Maria*
yum -y remove Maria*
rm -rf /var/lib/mysql/*
⚠️ 问题3：Windows初始化提示丢失VCRUNTIME140_1.dll

原因分析：此问题是缺少MySQL依赖的Windows库导致的。

解决方案：安装Visual C++ Redistributable for Visual Studio 2019，或者搜索"微软常用运行库合集2019"，下载并安装后即可正常安装MySQL。

📌 其他常见问题

✓权限问题：确保数据目录有正确读写权限
✓防火墙：确保允许3306端口访问

一、权限表

MySQL有5个常用的控制权限的表：user表、db表、tables_priv表、columns_priv表和procs_priv表。

📌 1. user表

存放账户信息及全局级别（所有数据库）权限，决定了来自哪些主机的哪些账户可以访问数据库实例，如果有全局权限则意味着对所有数据库都有此权限。

# 查看user表结构
mysql> USE mysql;
mysql> DESC user;

（1）账号字段

User：账户名称
Host：允许访问的客户端IP地址或主机地址，为"%"时表示所有客户端的账户
Host+User：user表的主键，用于区分MySQL中的账户

（2）权限字段

以"_priv"结尾的字段，保存账户的全局权限，如"Select_priv"查询权限、"Insert_priv"插入权限等。数据类型都是ENUM枚举类型，取值为N或Y，N表示无对应权限，Y则表示拥有该权限。

（3）安全连接字段

ssl_type：用于保存安全连接的类型
ssl_cipher：用于保存安全加密连接的特定密码
x509_issuer：保存由CA签发的有效的X509证书
x509_subject：保存包含主题的有效的X509证书

（4）资源限制字段

保存账户对可使用的服务器资源的限制
max_questions：每小时允许用户执行查询操作的最多次数
max_updates：每小时允许用户执行更新操作的最多次数
max_connections：每小时允许用户建立连接的最多次数
max_user_connections：允许单个用户同时建立连接的最多数量

（5）身份验证字段

plugin：用于指定账户的验证插件名称
authentication_string：根据对应的验证插件对明文密码加密后的字符串
password_expired：密码是否过期
password_last_changed：密码最后修改时间
password_lifetime：密码的有效期
Password_reuse_history：修改密码不允许与最近几次使用的密码重复
Password_reuse_time：修改密码不允许与最近多少天使用的密码重复
Password_require_current：修改密码是否需要提供当前的登录密码

（6）账户锁定字段

account_locked：保存当前账户是否是锁定状态，当为Y时，即账户锁定，不能用于连接到服务器。

三、权限管理
📌 （一）MySQL的各种权限

1. 数据权限

数据权限主要是针对数据库表中的数据行的操作权限，如SELECT、INSERT、UPDATE、DELETE等。

2. 结构权限

结构权限主要针对数据库表的结构或数据库的对象进行操作的权限，如CREATE、ALTER、DROP、INDEX等。

3. 管理权限

管理权限指的是通常只有管理员才有的管理权限，如SUPER、PROCESS、FILE、RELOAD等。

📌 （二）查看权限
📌 语法
SHOW GRANTS [FOR 账户]
📌 （三）授予权限
📌 语法
GRANT
    priv_type [(column_list)]
      [, priv_type [(column_list)]] ...
    ON [object_type] priv_level
    TO user_or_role [, user_or_role] ...
    [WITH GRANT OPTION]

# 参数说明：
# priv_type：权限类型
# column_list：字段列表用于设置列权限
# object_type：权限作用的目标类型，默认为TABLE，其值还可以是FUNCTION(函数)、PROCEDURE(存储过程)
# priv_level：权限级别
# user_or_role：账户或角色
# WITH GRANT OPTION：表示当前账户可以为其他账户进行授权
📌 示例
# 示例1：授予user1账户对test数据库中的student表的SELECT权限，以及对name和phone字段的插入权限
mysql> GRANT SELECT,INSERT(name,phone) ON test.student TO user1;

# 示例2：授予user1账户对test数据库所有的权限
mysql> GRANT ALL ON test.* TO user1;

# 查看授予的权限
mysql> SHOW GRANTS FOR user1;
📌 （四）收回权限
📌 语法
REVOKE
    priv_type [(column_list)]
      [, priv_type [(column_list)]] ...
    ON [object_type] priv_level
    FROM user_or_role [, user_or_role] ...
📌 示例
# 示例1：收回user1账户对test数据库student表的插入权限
mysql> REVOKE INSERT(name,phone) ON test.student FROM user1;

# 示例2：收回user1账户所有的权限
mysql> REVOKE ALL ON test.student FROM user1;

# 查看收回后的权限
mysql> SHOW GRANTS FOR user1;
📌 （五）刷新权限

💡 为什么要刷新权限？

刷新权限指的是从系统数据库mysql中的权限表中重新加载账户的权限。这是因为GRANT、CREATE USER等操作会将服务器的缓存信息保存到内存中，而REVOKE、DROP USER操作并不会同步到内存中。因此可能会造成服务器内存的消耗，所以在REVOKE、DROP USER操作后建议使用FLUSH PRIVILEGES重新加载账户的权限。

mysql> FLUSH PRIVILEGES;

  • user表：存放账户信息及全局级别权限，Host+User为主键
  • db表：存放数据库级别的权限
  • tables_priv表：存放表级别的权限
  • columns_priv表：存放列级别的权限

  • 删除账户：DROP USER 'user'@'host'

  • 权限类型：数据权限（SELECT、INSERT等）、结构权限（CREATE、DROP等）、管理权限（SUPER等）

  • 授予权限：GRANT priv_type ON priv_level TO user

📚 项目知识导入

在数据库运行过程中，一般都会产生很多的日志信息，这些日志信息记录着数据库运行工作的具体情况，帮助数据库管理员追踪在数据库运行期间发生过的各种事件。MySQL中的日志有6种，分别是错误日志、一般查询日志、慢查询日志、二进制日志、中继日志和DDL日志。

一、错误日志

错误日志记录MySQL服务在启动和停止时的正确及错误信息，还记录运行过程中的错误信息，是MySQL最重要的日志之一，当数据库由于出现任何故障导致无法正常使用时，可以首先查看此日志。

📌 配置错误日志组件

MySQL8.0的错误日志采用组件架构，通过不同的组件执行日志的写入和过滤功能。系统变量log_error_services指定过滤组件、写入组件。

-- 查看MySQL8.0默认的过滤组件和写入组件
mysql> SELECT @@GLOBAL.log_error_services;
+----------------------------------------+
| @@GLOBAL.log_error_services            |
+----------------------------------------+
| log_filter_internal; log_sink_internal |
+----------------------------------------+
📌 配置错误日志过滤

log_filter_internal

根据log_error_verbosity和log_error_suppression_list系统变量，结合日志事件优先级和错误代码进行错误日志过滤。内置并默认启用。

log_filter_dragnet

根据用户提供的规则，结合dragnet.log_error_filter_rules系统变量进行错误日志过滤。

📌 基于优先级的错误日志过滤

错误日志事件的优先级分别为ERROR、WARNING或INFORMATION。由log_error_verbosity系统变量控制服务器日志的详细程度。

# 在my.cnf文件中配置
[mysqld]
log_error_verbosity=1     # 仅错误消息（1=错误，2=错误和警告，3=错误、警告和信息）
📌 启动和设置错误日志
# 默认开启错误日志功能
# 错误日志存储在MySQL数据库的数据文件夹下，通常名称为hostname.err

# 在my.cnf文件中配置
log-error=/var/log/mysqld.log

# 如果不知道错误日志的位置，可以通过查看变量log_error来获得
mysql> SHOW VARIABLES LIKE 'log_error';
📌 查看和删除错误日志
# Linux下使用tail命令查看
tail -3 /var/log/mysqld.log

# 使用mysqladmin命令开启新的错误日志
mysqladmin -uroot -p flush-logs
二、一般查询日志

一般查询日志记录客户端的操作语句。

📌 启动和设置一般查询日志
-- 查看一般查询日志是否打开
mysql> SHOW VARIABLES LIKE 'general_log';

-- 启用一般查询日志
mysql> SET GLOBAL general_log = 'ON';

-- 查看日志文件路径
mysql> SHOW VARIABLES LIKE 'general_log_file';
📌 删除一般查询日志

一般查询日志文件属于文本文件，可以直接删除，删除后需要使用flush logs命令重新生成一般查询日志文件。

三、慢查询日志

慢查询日志用于记录在MySQL中执行时间超过指定时间的查询语句。MySQL的慢查询日志是排查问题SQL语句，以及检查当前MySQL性能的一个重要工具。

📌 启动和设置慢查询日志
# 在my.cnf文件中配置
[mysqld]
log-slow-queries=dir/filename
long_query_time=n

# 参数说明：
# dir：慢查询日志的存储路径，默认存储到MySQL的数据目录下
# filename：日志的文件名，完整名称为filename-slow.log
# n：设定的时间值（秒），默认时间为10秒
📌 查看慢查询日志

慢查询日志以文本文件形式存储，可以使用普通文本文件查看工具来查看。

Time

该查询发生的时间

User@Host

发起该查询的主机

Query_time

该查询总共用了多少时间

Lock_time

在查询过程中锁定表的时间

Rows_sent

返回了多少行数据

Rows_examined

表扫描了多少行数据才得到结果

📌 mysqldumpslow命令

庞大的慢查询日志文件采用人工分析很困难，可以使用MySQL自带的mysqldumpslow命令来分析慢查询日志。

# 得到记录次数最多的10条SQL语句
mysqldumpslow -s c -t 10 /var/lib/mysql/localhost-slow.log

# 得到按照查询时间排序的前10条含有左连接的SQL语句
mysqldumpslow -s t -t 10 -g "left join" /var/lib/mysql/localhost-slow.log
四、二进制日志

二进制日志用于记录所有的DDL语句和DML语句，但是不包括数据查询语句。语句以"事件"的形式保存，它描述了数据的更改过程。二进制日志对于"灾难"出现时的数据恢复具有极其重要的作用。

📌 启动和设置二进制日志
# 在my.cnf文件中配置
[mysqld]
log-bin=mysql-bin

# 查看当前二进制日志文件的位置和状态
mysql> SHOW MASTER STATUS;

# 查看日志文件列表
mysql> SHOW BINARY LOGS;
📌 使用mysqlbinlog查看日志
# 使用mysqlbinlog命令查看日志文件内容
mysqlbinlog [options] log-files1 log-files2...
📌 删除二进制日志
-- 方法1：reset master命令（删除所有二进制日志，新日志编号从000001开始）
mysql> RESET MASTER;

-- 方法2：purge master logs to命令（以日志编号为条件删除）
mysql> PURGE MASTER LOGS TO 'mysql-bin.000003';

-- 方法3：purge master logs before命令（以时间为条件删除）
mysql> PURGE MASTER LOGS BEFORE '2024-01-01 00:00:00';

-- 方法4：expire_logs_days配置项（设置日志过期天数）
# 在my.cnf文件中配置
expire_logs_days=7

  • 组件架构：MySQL8.0采用组件架构，包括过滤组件和写入组件
  • 过滤组件：log_filter_internal（默认）、log_filter_dragnet
  • 优先级：ERROR、WARNING、INFORMATION，由log_error_verbosity控制
  • 查看方法：tail -f /var/log/mysqld.log

三、慢查询日志

  • 作用：记录执行时间超过阈值的SQL语句，用于性能优化
  • 配置：slow_query_log=ON，long_query_time=阈值（秒）
  • 分析工具：mysqldumpslow，支持按次数、时间、记录数排序
  • 参数：-s排序方式，-t返回条数，-g正则匹配

四、二进制日志与数据恢复

  • 作用：记录DDL和DML语句，用于数据恢复和主从复制
  • 查看：SHOW MASTER STATUS、SHOW BINARY LOGS、mysqlbinlog
  • 基于时间点恢复：--start-datetime和--stop-datetime参数
  • 基于字节位置恢复：--start-position和--stop-position参数，更精确

📋 任务5-1 使用mysqldump备份和恢复数据

📖 任务背景

在MySQL数据库中自带了一款功能强大的逻辑备份工具——mysqldump。它不仅常常被用于执行数据备份任务，甚至可用于数据迁移。使用mysqldump命令可以把数据从MySQL库中以逻辑的SQL语句的形式直接输出或生成备份的文件。尽管mysqldump只是一个命令行工具，但是功能已经相当完善，是数据库管理员日常运维中最常用的备份工具之一。

🎯 任务目标

  • 掌握使用mysqldump命令备份整个MySQL数据库的方法
  • 学会使用mysqldump备份指定数据库及其中的事件和存储过程
  • 理解--single-transaction参数的作用和使用场景
  • 掌握使用mysql命令恢复备份数据的操作流程
🔧 步骤1：备份整个数据库

在MySQL数据库中，最常用的备份方式是使用mysqldump备份整个数据库。使用mysqldump命令时，需要指明备份数据库的账号密码，并设置备份文件的路径和文件名。该命令会将所有数据库的结构和数据导出为SQL语句，保存到指定的文件中。

# 备份所有数据库到full.sql文件
mysqldump -uroot -p --all-databases > full.sql

# 检查备份是否成功（查看文件末尾）
tail -10 full.sql
# 如果显示 "-- Dump completed on ..." 则表示备份成功
🔧 步骤2：备份指定数据库

除了备份整个数据库外，还可以通过设置参数来备份指定的数据库。使用--databases参数可以指定要备份的数据库名称。同时，--single-transaction参数使得在备份过程中保证InnoDB引擎表的一致性。-E参数表示导出事件（Events），-R参数表示导出存储过程（Routines）。

# 备份指定数据库（含事件和存储过程）
mysqldump -uroot -p --single-transaction -E -R --databases test > test.sql

# 参数说明：
# --single-transaction  保证InnoDB表的一致性
# -E  导出事件(Events)
# -R  导出存储过程(Routines)
🔧 步骤3：恢复数据

相对于备份数据，逻辑备份的恢复数据操作显得很简单。由于mysqldump生成的备份文件本质上是SQL语句的集合，因此只需要将备份文件导入数据库即可完成恢复。

# 恢复整个数据库
mysql -uroot -p < full.sql

# 恢复指定数据库
mysql -uroot -p test < test.sql

✓mysqldump是MySQL自带的逻辑备份工具
✓--single-transaction保证InnoDB表的一致性
✓-E参数导出事件，-R参数导出存储过程
✓备份文件本质是SQL语句的集合
✓恢复操作通过mysql命令导入SQL文件实现
✓备份成功标志：文件末尾显示Dump completed
📋 任务5-2 使用Percona XtraBackup备份和恢复数据

📖 任务背景

Percona XtraBackup是一个基于MySQL服务器的开源热备份实用程序，它不会在备份期间锁定数据库。无论是24×7高负载服务器还是低事务量环境，Percona XtraBackup都使备份成为一个无缝过程，而不会破坏生产环境中服务器的性能。与mysqldump的逻辑备份不同，XtraBackup进行的是物理备份，直接复制数据库的数据文件，因此在大数据量场景下备份和恢复速度更快。

🎯 任务目标

  • 掌握Percona XtraBackup的安装和配置方法
  • 学会使用xtrabackup命令进行MySQL数据库的热备份
  • 理解物理备份与逻辑备份的区别和适用场景
  • 掌握使用xtrabackup进行数据恢复的完整流程
🔧 步骤1：安装依赖包

在安装Percona XtraBackup之前，需要确保系统已安装相关的依赖软件包。

# 安装依赖包
yum -y install libev
yum -y install perl
yum -y install rsync perl perl-Digest-MD5
🔧 步骤2：安装Percona存储库

官方推荐从Percona存储库安装XtraBackup。首先需要安装Percona的YUM仓库配置文件。

# 安装Percona存储库
sudo yum install -y https://repo.percona.com/yum/percona-release-latest.noarch.rpm

# 启用Percona XtraBackup存储库
percona-release enable-only tools release
🔧 步骤3：安装Percona XtraBackup

Percona XtraBackup 8.0只支持MySQL 8.0版本。如果使用的是MySQL的其他版本（如5.7），则需要安装Percona XtraBackup 2.4版本。

# 安装Percona XtraBackup 8.0（适用于MySQL 8.0）
yum install -y percona-xtrabackup-80

# 如果是MySQL 5.7，安装2.4版本
yum install -y percona-xtrabackup-24
🔧 步骤4：赋予账户备份权限

在进行备份之前，需要确保用于备份的MySQL账户具有BACKUP_ADMIN权限。

# 授予root账户BACKUP_ADMIN权限
mysql> GRANT BACKUP_ADMIN ON *.* TO 'root'@'%';
mysql> FLUSH PRIVILEGES;
🔧 步骤5：备份数据

安装完成并配置好权限后，即可使用xtrabackup命令进行全库备份。当看到"completed OK!"字样时，表示本次物理备份成功完成。

# 全库备份
xtrabackup --defaults-file=/etc/my.cnf -uroot -p --backup --target-dir=/XtraBackup/backup

# 备份成功输出：completed OK!
🔧 步骤6：恢复数据

使用--backup参数进行备份后，需要先执行"准备"（prepare）操作才能进行恢复。这是因为备份时数据文件是在不同时间点复制的，文件之间存在时间不一致的问题。准备完成后，即可将备份文件恢复到MySQL的数据目录中。需要注意的是，恢复时必须确保目标数据目录为空。

# 步骤6a：准备备份（使数据文件一致）
xtrabackup --defaults-file=/etc/my.cnf -uroot -p --prepare --target-dir=/XtraBackup/backup

# 步骤6b：恢复数据（确保数据目录为空）
xtrabackup --defaults-file=/etc/my.cnf -uroot -p --copy-back --target-dir=/XtraBackup/backup
💡 知识要点
✓Percona XtraBackup是开源的物理热备份工具
✓物理备份比逻辑备份速度更快，适合大数据量
✓备份过程不需要锁定数据库，不影响业务
✓需要BACKUP_ADMIN权限才能执行热备份
✓恢复前必须先执行prepare操作保证数据一致
✓恢复时目标数据目录必须为空
📋 任务5-3 使用mysqldump+crontab自动备份数据库

📖 任务背景

在实际的项目场景中，数据库备份是一项需要定期执行的运维任务。手动执行备份命令不仅效率低下，而且容易遗忘，一旦服务器发生故障而没有及时备份，将造成不可挽回的数据损失。因此，实现数据库的自动备份是每个数据库管理员必须掌握的技能。在Linux系统下，可以通过crontab命令来设置定时任务，结合mysqldump备份脚本，实现数据库的自动化备份管理。

🎯 任务目标

  • 学会编写MySQL数据库自动备份的Shell脚本
  • 掌握crontab定时任务的配置方法
  • 理解备份轮转策略（保留最近N天的备份）
  • 能够实现完整的数据库自动备份解决方案
🔧 步骤1：编写备份脚本文件

首先需要创建一个Shell脚本文件来执行备份操作。该脚本实现了以下功能：设置备份文件保留天数（默认31天）、指定备份保存路径、使用日期时间作为备份文件名的一部分、执行mysqldump备份命令、记录备份日志、自动删除超过保留天数的旧备份文件。

#!/bin/bash
# 保存备份个数，备份31天数据
number=31
# 备份保存路径
backup_dir=/root/mysqlbackup
# 日期（用于文件名）
dd=`date +%Y-%m-%d-%H-%M-%S`
# 备份工具
tool=mysqldump
# 数据库用户名
username=root
# 数据库密码
password=Hello123.
# 要备份的数据库名
database_name=mydb

# 如果备份目录不存在则创建
if [ ! -d $backup_dir ]; then
    mkdir -p $backup_dir;
fi

# 执行备份
$tool -u $username -p$password $database_name > $backup_dir/$database_name-$dd.sql

# 写创建备份日志
echo "create $backup_dir/$database_name-$dd.dupm" >> $backup_dir/log.txt

# 找出需要删除的备份
delfile=`ls -l -crt $backup_dir/*.sql | awk '{print $9 }' | head -1`

# 判断备份数量是否大于保留数
count=`ls -l -crt $backup_dir/*.sql | awk '{print $9 }' | wc -l`
if [ $count -gt $number ]
then
    # 删除最早生成的备份，只保留number数量的备份
    rm $delfile
    # 写删除文件日志
    echo "delete $delfile" >> $backup_dir/log.txt
fi

📋 任务5-4 迁移MySQL数据库

📖 任务背景

在项目场景中，有时需要将MySQL数据库从一台服务器迁移到另一台服务器，或者从一个操作系统迁移到另一个操作系统。这种跨系统、跨主机的数据库迁移是数据库运维中常见的需求。本任务要求将Linux系统中的MySQL数据库迁移到Windows系统的MySQL数据库中，实现不同系统下的相同数据库产品间的数据库迁移工作。

🔧 步骤1：允许远程访问

在进行跨主机数据库迁移之前，首先需要确保源数据库（Linux系统）允许远程主机访问。默认情况下，MySQL的root账户只允许本地登录。需要修改mysql.user表中root用户的host字段为"%"，表示允许任何主机远程连接。

# 允许root账户远程访问

mysql> FLUSH PRIVILEGES;

使用mysqldump命令导入数据库时，要求目标数据库必须先存在。因此在导入数据之前，需要在目标主机（Windows系统）上先创建好同名的目标数据库。

# 创建目标数据库

🔧 步骤3：执行数据库迁移

迁移的核心操作是在目标主机上使用mysqldump命令从远程源数据库读取数据，然后通过管道直接导入到本地数据库中。这种方式不需要生成中间文件，数据直接在网络上传输。

# 在Windows目标主机上执行迁移命令
mysqldump -h192.168.226.100 -uroot -pHello123. sakila | mysql -uroot -p123456 sakila

# 命令说明：
# -h192.168.226.100  指定源数据库IP地址
# | 管道符，将mysqldump输出直接传给mysql

# 在目标主机上验证迁移结果

mysql> USE sakila;
mysql> SHOW TABLES;

# 如果显示所有表，则迁移成功

  • 权限要求：需要BACKUP_ADMIN权限

四、自动备份

  • 脚本：Shell脚本封装mysqldump命令，设置备份路径、保留天数等变量
  • 定时：crontab配置定时任务，格式：分 时 日 月 周 命令
  • 保留策略：建议保留31天备份，自动删除过期文件并记录日志

五、数据库迁移

  • 跨主机迁移：mysqldump -h源IP -uroot -p密码 库名 | mysql -uroot -p密码 库名
  • 前提条件：源数据库开启远程访问（host='%'），目标数据库预先创建
  • 表数据迁移：通过导出文本文件（SELECT INTO OUTFILE）作为中间格式，再导入目标数据库
  • 导入方式：LOAD DATA INFILE和mysqlimport都可导入文本文件

🔧 步骤1：调整innodb_buffer_pool_size参数

innodb_buffer_pool_size参数决定缓冲池大小。参数默认为128MB，官方建议将该参数调整为操作系统物理内存40%~80%。如物理内存10G，可将该参数调整为6G。

# 查看当前缓冲池大小
mysql> SHOW GLOBAL VARIABLES LIKE '%innodb_buffer_pool_size%';

+-------------------------+-----------+
| Variable_name           | Value     |
+-------------------------+-----------+
| innodb_buffer_pool_size | 134217728 |
+-------------------------+-----------+

# 调整为6GB（6442450944字节）
mysql> SET GLOBAL innodb_buffer_pool_size=6442450944;
🔧 步骤2：调整tmp_table_size参数

查看自启动以来，涉及临时表的使用情况。如果Created_tmp_disk_tables / Created_tmp_tables的比值过高（超过25%），说明临时表频繁创建到磁盘上，需要增加tmp_table_size参数。

# 查看临时表使用情况
mysql> SHOW GLOBAL STATUS WHERE variable_name IN ('Created_tmp_disk_tables','Created_tmp_tables');

+-------------------------+--------+
| Variable_name           | Value  |
+-------------------------+--------+
| Created_tmp_disk_tables | 712092 |
| Created_tmp_tables      | 726359 |
+-------------------------+--------+

# 计算比值：712092/726359*100% = 98%（过高）

# 查看当前tmp_table_size
mysql> SHOW GLOBAL VARIABLES LIKE '%tmp_table_size%';

# 调整tmp_table_size为24MB
mysql> SET GLOBAL tmp_table_size=25165824;

✓建议设置为物理内存的40%~80%
✓tmp_table_size控制临时表大小
✓临时表磁盘创建比值应低于25%

🔧 步骤3：分析查询

查询category_id为1且comments大于1的情况下，views最多的article_id。

mysql> EXPLAIN SELECT author_id
    FROM `article`
    WHERE category_id = 1 AND comments > 1
    ORDER BY views DESC
    LIMIT 1;

# 分析结果：
# type: ALL（全表扫描，最差）
# Extra: Using where; Using filesort（文件排序，最差）
🔧 步骤4：优化查询

根据查询条件建立联合索引，但由于comments使用范围查询，会导致索引失效。需要调整索引策略。

# 分析发现：type变为range，但仍有Using filesort

# 删除旧索引
mysql> DROP INDEX x ON article;

# 建立新索引（去掉comments）
mysql> ALTER TABLE `article` ADD INDEX y (`category_id`, `views`);

# 再次分析：type变为ref，优化成功

✓type字段表示访问类型（ALL最差，const最好）

📖 任务背景

企业为了搭建高可用的数据库服务，复制是一种常用的技术方法。MySQL复制是指从一个MySQL主服务器（master）将数据拷贝到另一台或多台MySQL服务器（slave）的过程。本任务要求在Linux系统下，基于二进制日志事件，建立MySQL主从复制并测试其同步情况。

📋 任务7-2 配置MySQL半同步复制

📋 任务7-4 基于GTID建立MySQL主从复制

📖 任务背景

GTID（Global Transaction Identifier，全局事务标识符）比传统复制的优势在于搭建过程更简单、复制过程更安全、故障转移更优。在数据库事务提交时会在binlog中产生一个对应的GTID，从数据库会通过GTID来确定同步的位置，不用再去找二进制日志文件的复制位置。

🎯 任务目标

  • 理解GTID复制的概念和优势
  • 掌握GTID复制的配置方法
  • 学会使用GTID进行主从复制
🔧 步骤1：开启GTID

# 编辑配置文件（主库和从库都要配置）
vi /etc/my.cnf

[mysqld]
gtid_mode=on                      # 开启GTID
log_slave_updates=on
enforce_gtid_consistency=true    # 保证GTID一致性
🔧 步骤2：重启服务并验证

分别在主库和从库重启MySQL服务后，查看gtid_mode状态。

# 重启服务

+---------------+-------+

+---------------+-------+

# 在主库执行

  • 核心组件：IO线程（拉取日志）、SQL线程（应用日志）

二、主从复制配置

  • 前提条件：开启二进制日志、配置唯一server_id
  • 配置流程：准备实例 → 复制数据 → 配置server_id → 创建复制账户 → 配置连接 → 启动复制
  • 状态监控：Slave_IO_Running、Slave_SQL_Running都为Yes表示正常
  • 延迟查看：Seconds_Behind_Master越小越好

三、半同步复制

  • 原理：Master等待至少一个Slave接收事务后才返回
  • 插件：Master端semisync_master.so，Slave端semisync_slave.so
  • 超时：超时后自动降级为异步复制
  • 持久化：需写入my.cnf配置文件

四、并行复制

  • 目的：解决SQL线程单线程瓶颈，减少主从延迟
  • 配置：slave_parallel_type=LOGICAL_CLOCK，slave_parallel_workers=4
  • 监控：通过performance_schema查看复制状态

五、GTID复制

  • GTID：全局事务标识符，自动标识唯一事务
  • 优势：搭建更简单、复制更安全、故障转移更优
  • 配置：gtid_mode=on，enforce_gtid_consistency=true
  • 使用：master_auto_position=1启用GTID自动定位

🚀 知识准备
📌 1. MySQL Cluster概述

MySQL Cluster是一套完整的高可用性、高性能的集群方案，由一组计算机构成，适合于分布式计算环境。MySQL Cluster实际上是在无共享的架构中实现的内存数据库集群，无共享的架构使得系统可以在廉价的硬件上运行，而且硬件和软件不会有单点故障。MySQL Cluster将标准的MySQL服务器与NDB Cluster存储引擎整合在一起。

📌 2. MySQL Cluster体系结构

MySQL Cluster架构包含如下4类节点：管理节点（MGM）、数据节点（NDB）、SQL节点（API）和客户端节点。

MGM

管理节点

负责管理集群内其他节点，提供配置数据、启动停止节点、运行备份等。配置文件为config.ini。

NDB

数据节点

用于保存MySQL Cluster的数据。数据节点数目与副本数目相关。配置文件为my.cnf。

API

SQL节点

提供访问MySQL Cluster数据的接口。配置文件为my.cnf。

👤

客户端节点

使用MySQL Cluster访问数据的节点，可以是MySQL客户端或支持Cluster API的应用程序。

📌 3. NDB存储引擎

MySQL Cluster使用NDB（Network DataBase）存储引擎。NDB存储引擎是一个内存存储引擎，所有数据（包括索引）都存放在内存中。NDB存储引擎支持事务、外键、行级锁等特性，但不支持全文索引。NDB存储引擎使用分布式架构，数据被分片存储在多个数据节点上，每个数据节点保存完整数据的一个片段。

🔧 步骤1：安装MySQL Cluster

从MySQL官网下载Linux版本的MySQL Cluster安装包，上传到各服务器后解压安装。

# 创建mysql用户和组，并设置权限

chown -R mysql:mysql /usr/local/mysql
🔧 步骤2：配置管理节点

在管理节点上创建配置文件目录和config.ini配置文件。

# 创建配置文件目录
mkdir /var/lib/mysql-cluster

创建管理节点配置文件config.ini，内容如下：

# /var/lib/mysql-cluster/config.ini
[ndbd default]
NoOfReplicas=2
DataMemory=80M
IndexMemory=18M

[ndb_mgmd]
HostName=192.168.131.128
DataDir=/var/lib/mysql-cluster

[ndbd]
HostName=192.168.131.128
DataDir=/usr/local/mysql/data

[ndbd]
HostName=192.168.131.129
DataDir=/usr/local/mysql/data

[mysqld]
HostName=192.168.131.128

[mysqld]
HostName=192.168.131.129

📌 config.ini 核心参数说明

  • NoOfReplicas：数据副本数量，设置为2表示每个数据节点都有一个镜像节点
  • DataMemory：数据节点用于存储数据的内存大小
  • IndexMemory：数据节点用于存储索引的内存大小
  • HostName：各节点的IP地址
  • DataDir：数据存储目录

cd /usr/local/mysql

🔧 步骤3：配置数据节点和SQL节点

在数据节点和SQL节点上创建配置文件my.cnf：

# /etc/my.cnf
[mysqld]
basedir=/usr/local/mysql
datadir=/usr/local/mysql/data
ndbcluster
ndb-connectstring=192.168.131.128:1186

[mysql_cluster]
ndb-connectstring=192.168.131.128:1186

# 初始化MySQL数据库
cd /usr/local/mysql
./scripts/mysql_install_db --user=mysql

# 启动数据节点
./bin/ndbd --initial

# 启动SQL节点
./bin/mysqld_safe --user=mysql &
🔧 步骤4：验证MySQL Cluster状态
# 登录管理节点查看集群状态
./bin/ndb_mgm
ndb_mgm> SHOW

# 确认所有节点连接正常，状态为connected

📌 启动顺序

必须按顺序启动：管理节点 → 数据节点 → SQL节点。先启动ndb_mgmd，再启动ndbd，最后启动mysqld。

🔧 常见问题解决
1启动管理节点时出现：Unable to bind port 1186

原因分析

端口1186被其他程序占用。

解决方案

查找占用端口的进程并结束，或修改config.ini中的端口号。

netstat -anp | grep 1186
2数据节点无法连接到管理节点：Could not connect to management server

原因分析

防火墙阻止了连接，或管理节点未启动。

解决方案

确认管理节点已启动；关闭防火墙或开放1186端口；检查config.ini中的IP地址配置。

service iptables stop
3SQL节点无法启动：Can't connect to NDB Cluster

原因分析

数据节点未启动，或SQL节点配置错误。

解决方案

确认数据节点已正常启动；检查my.cnf中的ndb-connectstring配置；查看错误日志排查问题。

4创建表时提示：Unknown storage engine 'ndbcluster'

原因分析

SQL节点未加载ndbcluster存储引擎。

解决方案

确认SQL节点启动时使用了--ndbcluster参数；在my.cnf中确保包含ndbcluster配置项。

5数据节点内存不足：Out of memory

原因分析

DataMemory或IndexMemory设置过大。

解决方案

调整config.ini中的内存参数，或增加物理内存/增加数据节点数量。

# 调整内存参数示例
DataMemory=50M
IndexMemory=10M

解析注释中的远程数据库配置并建立连接；检查二进制日志是否开启；判断语法开始位置（必须以inception_magic_start开始）；逐行解析并进行语法解析；进行建表的各项校验（表名、列名长度、关键字、存储引擎、字符集等）

2

执行阶段

DDL语句和DML语句处理方式不同：DML通过二进制日志解析实现回滚，DDL根据语法规则直接生成逆向SQL。记录二进制日志位置，执行失败时记录失败原因并结束执行操作。

3

备份阶段

只备份执行成功的语句。DDL的备份是保存自动生成的逆向SQL语句；DML的备份是根据执行前后记录的二进制日志位置和线程号，以模拟从库的形式获取二进制日志信息。解析二进制日志要求必须为ROW模式。

以美创科技的数据库运行安全管理平台为例，商用数据库运维平台通常包含以下核心功能：

聚合大屏

巡检方法

包括上下班巡检和全面巡检。上下班巡检根据设定的时间，从可用性、资源就绪、可靠性、变化和作业等多个维度完成自动巡检。全面巡检从8个维度实现数据库的巡检分析工作。

性能诊断

全面分析数据库各资源使用情况，并给出数据库性能是否正常的结论。支持对每个指标项以图表方式直观查看，同时支持下钻查看详情与历史功能。

容量预警

让运维人员快速了解当前剩余表空间能否支撑未来一段时间内的数据增长量。针对容量问题提供相应的告警与处理方法，例如表空间扩容、归档日志清理等。

SQL审核

展示选定时间段内的低效SQL，并给出SQL的DB Time消耗值分布趋势图。根据审核规则的重要程度不同，分为严重、告警、提醒三大类。

自动化部署

只需完成基本的主机信息和数据库信息配置，即可实现一键自动化批量部署数据库（包括各类集群），极大地减轻运维工作量，提升运维效率。

智能编排

通过模板来定义执行任务、执行顺序、执行输入和输出，然后通过执行模板来完成任务，实现全自动自动化问题处理，同时降低运维成本并提高效率。

# t_goinception.py
import pymysql

# 连接goInception
# 注意：这里的host和port是goInception的地址，不是MySQL的地址
conn = pymysql.connect(
    host='127.0.0.1',
    port=4000,
    user='',
    password='',
    charset='utf8mb4'
)

cursor = conn.cursor()

# 定义要执行的SQL语句
# 注意：SQL语句前需要加上注释指定目标数据库连接信息
sql = """
/*--user=root;--password=123456;--host=127.0.0.1;--port=3306;*/
inception_magic_start;
use test_inc;
create table t1(id int primary key, c1 int);
insert into t1(id, c1, c2) values(1, 1, 1);
inception_magic_commit;
"""

# 执行SQL
cursor.execute(sql)

# 获取结果
result = cursor.fetchall()

# 打印结果
from prettytable import PrettyTable
table = PrettyTable()
table.field_names = ['ID', 'stage', 'errlevel', 'stagestatus', 'errormessage', 'SQL', 'Affected_rows', 'sequence', 'backup_dbname', 'execute_time']

for row in result:
    table.add_row(row)

print(table)

cursor.close()
conn.close()
🔧 步骤5：调用Python脚本

📌 审核结果说明

执行结果显示，goInception对Python脚本中的DDL语句进行了审核，并检查出在向t1表中插入记录时，"c2字段不存在"的错误。将Python脚本中的create table t1(id int primary key,c1 int);语句改为create table t1(id int primary key,c1 int,c2 int);，即建立t1表的c2字段，再执行一次即可通过审核。
