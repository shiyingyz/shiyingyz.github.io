Database Operations & Management

项目六：监控、测试并优化 MySQL 性能

项目小结与自我检测

📚 项目总结

核心知识回顾

性能监控基础

  • QPS（Queries Per Second）：每秒查询数，反映数据库查询处理能力
  • TPS（Transactions Per Second）：每秒事务数，反映数据库事务处理能力
  • 缓冲池命中率：InnoDB 缓冲池命中比例，理想值 > 99%

监控工具使用

  • MONyog：图形化 MySQL 监控，Dashboard 查看 QPS / 连接数 / 缓冲池命中率
  • mysqladmin：命令行工具，status / extended-status 命令

压力测试

  • Sysbench 三阶段：prepare（准备数据）→ run（执行压测）→ cleanup（清理数据）
  • 关键指标：TPS、QPS、平均延迟、95% 分位延迟
  • 注意修改并发线程数，多次测试取平均值

EXPLAIN 执行计划分析

  • type 字段等级（从差到好）：ALL → index → range → ref → eq_ref → const
  • key 字段：实际使用的索引
  • rows 字段：预估扫描行数，越小越好
  • Extra 字段：Using filesort（需要优化）/ Using index（覆盖索引，好现象）

索引优化原则

  • 联合索引遵循最左前缀原则
  • 范围查询（> < >= <= BETWEEN LIKE）会阻断后续索引
  • LEFT JOIN 优化右表、RIGHT JOIN 优化左表
  • 避免在索引列上做运算或使用函数

参数调优

  • innodb_buffer_pool_size：调整为物理内存 40% ~ 80%
  • tmp_table_size：监控 Created_tmp_disk_tables / Created_tmp_tables 比例 < 25%
  • long_query_time：设置慢查询阈值（建议 2-3 秒）

云数据库运维

  • 阿里云 RDS 申请流程
  • 白名单设置、外网地址申请
  • 慢查询日志开启与分析

常见问题排查清单

问题现象 可能原因 排查方法
查询缓慢 缺少索引 / 全表扫描 EXPLAIN 分析 type 字段
系统假死 锁等待 / 长事务 SHOW PROCESSLIST 查看
连接数满 max_connections 不足 调整参数或优化连接池
磁盘 IO 高 缓冲池太小 加大 innodb_buffer_pool_size
临时表过多 tmp_table_size 太小 监控比例，适当调大
ERROR 2013 连接超时 检查 wait_timeout 参数

命令速查表

监控命令
SHOW STATUS LIKE 'QPS%';
SHOW STATUS LIKE 'Threads%';
SHOW GLOBAL VARIABLES LIKE 'innodb_buffer_pool_size%';
SHOW GLOBAL STATUS WHERE variable_name IN ('Created_tmp_disk_tables','Created_tmp_tables');

索引操作

压测命令
sysbench --mysql-host=IP --mysql-port=PORT --mysql-user=USER --mysql-password=PASS \
  --mysql-db=DB --table_size=25000 --tables=10 --threads=8 --time=60 \

📝 自我检测
Q1.EXPLAIN 结果中 type=ALL 和 type=ref 的含义是什么？
A ALL 表示索引扫描，ref 表示全表扫描
B ALL 表示全表扫描，ref 表示通过索引引用
C ALL 和 ref 都是索引扫描
D ALL 表示只使用缓存，ref 表示使用临时表
✓ 解析：
type=ALL 表示全表扫描（性能最差），type=ref 表示通过索引列引用查找（性能较好）。type 字段值从差到好排序为：ALL → index → range → ref → eq_ref → const。
Q2.压力测试中 TPS 和 QPS 的区别是什么？
A TPS 是每秒事务数，QPS 是每秒查询数
B TPS 是每秒查询数，QPS 是每秒事务数
C 两者完全相同
D TPS 只用于读操作，QPS 只用于写操作
✓ 解析：
TPS（Transactions Per Second）是每秒事务数，一个事务可能包含多条 SQL。QPS（Queries Per Second）是每秒查询数，衡量单条 SQL 的执行频率。
Q3.索引是越多越好吗？
A 是的，索引越多查询越快
B 不是，索引会增加写入开销和存储空间，需要权衡
C 取决于表的大小，小表不需要索引
D 只有主键索引有用
✓ 解析：
索引虽然可以加快查询速度，但会增加 INSERT / UPDATE / DELETE 的开销，占用额外存储空间。应在频繁查询的字段上创建索引，避免在频繁更新的字段上建过多索引。
Q4.联合索引 (category_id, comments, views) 中，查询条件 WHERE category_id=1 AND comments>1 ORDER BY views DESC，为什么 views 字段无法使用索引？
A views 字段类型不匹配
B comments>1 是范围查询，阻断了后续索引字段的使用
C ORDER BY 不能使用索引
D 联合索引最多只能使用前两个字段
✓ 解析：
根据 B+Tree 索引的最左前缀原则，当 comments 使用范围查询（>）时，其后的 views 字段无法再利用索引排序，导致 EXPLAIN 的 Extra 中出现 Using filesort。解决方案是去掉 comments，建立 (category_id, views) 联合索引。
Q5.LEFT JOIN 优化时应该优先为哪个表建索引？
A 左表
B 右表
C 两个表都建索引
D 行数少的表
✓ 解析：
LEFT JOIN 条件用于确定如何从右表搜索行，左边一定都有数据，所以右表是优化的关键点，应该优先为右表建立索引。同理，RIGHT JOIN 优化左表。
Q6.MySQL 默认的慢查询日志状态是？
A 开启，阈值 1 秒
B 开启，阈值 10 秒
C 关闭
D 开启，阈值 5 秒
✓ 解析：
MySQL 默认不启动慢查询日志，需要手动设置 SET GLOBAL slow_query_log=ON 来开启。默认的 long_query_time 阈值为 10 秒。
1.请写出 EXPLAIN type 字段从差到好的排列顺序。
📝 答案：

ALL → index → range → ref → eq_ref → const → system

2.如果 Created_tmp_disk_tables / Created_tmp_tables 的比例为 98%，说明什么问题？应该如何解决？
📝 答案：

说明临时表大小经常超过 tmp_table_size，大部分临时表被创建到磁盘上，影响 SQL 执行效率。应该适当增加 tmp_table_size 参数的值，使该比例降到 25% 以下。

3.描述使用 Sysbench 进行压力测试的三个阶段及其作用。
📝 答案：

prepare 阶段：准备测试数据，创建指定数量和大小的表；
run 阶段：执行压力测试，模拟并发读写操作，记录 TPS / QPS / 延迟等指标；
cleanup 阶段：清理测试数据，删除之前创建的表。
